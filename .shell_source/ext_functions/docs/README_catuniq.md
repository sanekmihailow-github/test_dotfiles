# catuniq - Фильтрация и анализ уникальных логов

## Синтаксис

```
catuniq [DURATION] [OPTIONS] [FILE...]
```

## Основные возможности

- **Временная фильтрация**: показывает только логи за указанный период (последние 5 минут по умолчанию)
- **Удаление дубликатов**: выводит только уникальные строки
- **Удаление и замена текста**: удаление переменных частей (--del) или замена паттернов на текст (--replace)
- **Условное удаление столбцов**: удаление столбца только если он совпадает с паттерном (--remove)
- **Условное изменение столбцов**: изменение значения столбца если он совпадает с паттерном (--change)
- **Подсчет частоты**: показывает сколько раз встречается каждая уникальная строка
- **Сортировка**: по алфавиту, частоте или времени появления
- **Поддержка любых форматов дат**: через regex паттерны
- **Улучшенная поддержка syslog**: автоматическое определение года для дат без года
- **Множественные колонки**: извлечение даты из нескольких колонок (например, `Dec 25 07:44:17` → колонки 1-3)

## Быстрый старт

```bash
# Показать уникальные логи за последние 15 минут
catuniq 15m /var/log/nginx/error.log

# Подсчитать частоту уникальных паттернов и отсортировать
catuniq 1h --count --sort freq /var/log/app.log

# Удалить IP адреса перед определением уникальности
catuniq 30m --del '[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}' /var/log/nginx/access.log

# Работа с syslog (дата разбита на 3 колонки)
catuniq 1h --col-start 1-3 --date-fmt='[A-Z][a-z]{2,4\} [0-9]{1,2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}' /var/log/syslog

# Анализ старых syslog логов с явным указанием года
catuniq --date-from "Dec 25 07:44:17" --year 2024 1h --col-start 1-3 /var/log/syslog.2024
```

### DURATION (опционально)

Первый позиционный аргумент может быть длительностью:
- `15s` - 15 секунд
- `5m` - 5 минут (по умолчанию)
- `2h` - 2 часа
- `1d` - 1 день

### OPTIONS

#### Опции даты/времени

- `--date-fmt=REGEX` - Regex паттерн для поиска даты в логах (по умолчанию: nginx формат)
- `--date-time=DURATION` - Временное окно для фильтрации (по умолчанию: 5m)
- `--date-from=DATETIME` - Начальное время для фильтрации (меняет логику окна, см. ниже)
- `--year=YYYY` - Явное указание года для syslog дат без года (например, 2025). Отключает автоматическое определение года
- `--col-start=SPEC` - Колонка(и) с датой (по умолчанию: 1)
  - Одна колонка: `--col-start 1`
  - Список колонок: `--col-start 1,2,3`
  - Диапазон колонок: `--col-start 1-3`
  - Смешанный формат: `--col-start 1,3-5,7`
  - Поиск по всей строке: `--col-start none`

**Логика временного окна:**

Когда `--date-from` **НЕ указан**:
```
Диапазон: [now - date-time, now]
Пример: --date-time 5m → "последние 5 минут до текущего момента"
```

Когда `--date-from` **УКАЗАН**:
```
Диапазон: [date-from, date-from + date-time]
Пример: --date-from "2026-01-20 20:00:00" --date-time 5m
       → "5 минут начиная с 20:00:00" (20:00:00 до 20:05:00)
```

**Поддерживаемые форматы для --date-from:**
```bash
# ISO 8601 (стандартный)
--date-from "2026-01-20 20:55:00"
--date-from "2026-01-20T20:55:00Z"

# Nginx формат (автоматически нормализуется)
--date-from "20/Jan/2026:20:55:00 +0300"
--date-from "[20/Jan/2026:20:55:00 +0300]"  # С квадратными скобками

# Syslog формат БЕЗ года (автоматически добавляется текущий/прошлый год)
--date-from "Dec 25 07:44:17"  # Автоопределение года
--date-from "Dec 25 07:44:17" --year 2025  # Явное указание года

# Другие понятные для GNU date форматы
--date-from "20 Jan 2026 20:55:00"
--date-from "Jan 20 20:55:00 2026"
```

**Автоматическое определение года для syslog:**

Когда `--date-from` содержит дату без года (например, "Dec 25 07:44:17"):
- Если `--year` указан - использует его
- Если `--year` НЕ указан - автоматически пробует текущий год, год-1, год-2 и выбирает ближайший к текущему моменту
- Это позволяет работать с syslog логами без ручного указания года

#### Опции обработки

- `--delimiter=CHAR` - Разделитель полей (по умолчанию: пробел)
- `--cut=SPEC` - Удалить колонки (например: 1,3,5 или 1-3,5)
- `--del=PATTERN` - Regex паттерн для удаления совпадающего текста (можно использовать несколько раз)
- `--replace=PATTERN===>REPLACEMENT` - Заменить regex паттерн на строку (формат: 'паттерн===>замена', можно использовать несколько раз)
- `--remove=COL:PATTERN` - Удалить колонку если она совпадает с паттерном (формат: 'номер_колонки:паттерн', можно использовать несколько раз)
- `--change=COL:PATTERN:TEXT` - Изменить значение колонки если она совпадает с паттерном (формат: 'номер_колонки:паттерн:новое_значение', экранируйте `:` как `\:` в
паттерне, можно использовать несколько раз)
- `--case-insensitive` - Игнорировать регистр при определении уникальности

#### Опции вывода

- `--count` - Показать количество вхождений каждой уникальной строки
- `--invert` - Показать только дубликаты (строки, появляющиеся 2+ раз)
- `--sort=MODE` - Сортировка: none, alpha, freq, time
- `--max-lines=N` - Ограничить вывод N строками
- `--output=FILE` - Записать вывод в файл вместо stdout

#### Прочее

- `--verbose` - Включить подробный отладочный вывод
- `--dry-run` - Показать конфигурацию без обработки
- `--help` - Показать справку

## Примеры использования

### 1. Nginx логи - найти уникальные типы ошибок

```bash
catuniq 1h \
    --col-start none \
    --del '[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}' \
    --del '"GET [^"]*' \
    --del '"POST [^"]*' \
    --count --sort freq \
    /var/log/nginx/error.log | head -20
```

**Результат:**
```
42    connect() failed (111: Connection refused)
38    upstream timed out
15    no live upstreams
```

### 2. Application логи с JSON

```bash
catuniq 1h \
    --col-start none \
    --date-fmt='[0-9]{4\}-[0-9]{2\}-[0-9]{2\}T[0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --del '"user":"[^"]*"' \
    --del '"request_id":"[^"]*"' \
    --count --sort freq \
    /var/log/app/errors.json
```

**Результат:**
```
125   {"level":"ERROR","error":"db_timeout"}
45    {"level":"ERROR","error":"auth_failed"}
12    {"level":"ERROR","error":"invalid_input"}
```

### 3. K3s/Kubernetes логи

```bash
catuniq 30m \
    --col-start none \
    --date-fmt='[0-9]{4\}-[0-9]{2\}-[0-9]{2\}T[0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --del 'pod=[^ ]+' \
    --del 'namespace=[^ ]+' \
    --del 'node=[^ ]+' \
    --del '[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}' \
    --count --sort freq \
    /var/log/k3s/agent.log | grep -i error
```

### 4. Syslog формат

```bash
# Syslog обычно использует формат: Dec 25 07:44:17 (дата разбита на колонки 1-3)

# Вариант 1: Поиск по всей строке (универсально, но медленнее)
catuniq 1h \
    --col-start none \
    --date-fmt='[A-Z][a-z]{2,4\} [0-9]{1,2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --count \
    /var/log/syslog

# Вариант 2: Указать колонки 1-3 (быстрее)
catuniq 1h \
    --col-start 1-3 \
    --date-fmt='[A-Z][a-z]{2,4\} [0-9]{1,2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --count \
    /var/log/syslog

# Вариант 3: Использование --date-from с автоопределением года
catuniq --date-from "Dec 25 07:44:17" 1h \
    --col-start 1-3 \
    --date-fmt='[A-Z][a-z]{2,4\} [0-9]{1,2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    /var/log/syslog

# Вариант 4: Явное указание года для старых логов
catuniq --date-from "Dec 25 07:44:17" --year 2025 1h \
    --col-start 1,2,3 \
    --date-fmt='[A-Z][a-z]{2,4\} [0-9]{1,2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    /var/log/syslog.2025
```

**Почему col-start 1-3 для syslog?**
Syslog разбивает дату на 3 колонки:
- Колонка 1: `Dec`
- Колонка 2: `25`
- Колонка 3: `07:44:17`

Указав `--col-start 1-3`, catuniq объединит их: `Dec 25 07:44:17` и применит regex.

### 5. Мониторинг в реальном времени

```bash
# Следить за новыми ошибками каждую минуту
watch -n 60 'catuniq 1m --count --sort freq /var/log/app/error.log'
```

### 6. Комбинирование с другими инструментами

```bash
# Найти уникальные ошибки и отправить email
catuniq 1h --count --sort freq /var/log/app/error.log | \
    head -10 | \
    mail -s "Top errors in last hour" admin@example.com

# Фильтровать по ключевому слову после удаления дубликатов
catuniq 1h /var/log/app.log | grep "database"

# Обработать несколько файлов логов
catuniq 1h /var/log/app/*.log | sort | uniq -c
```

### 7. Использование --date-from для анализа исторических логов

```bash
# Анализ логов за конкретный период (ISO формат)
catuniq 5m \
    --date-from "2026-01-20 14:30:00" \
    --count --sort freq \
    /var/log/app.log
# Показывает уникальные события с 14:30:00 до 14:35:00

# Анализ логов с точностью до секунды
catuniq 10s \
    --date-from "2026-01-20 14:30:00" \
    --col-start none \
    --del 'request_id=[^ ]+' \
    /var/log/app.log
# Показывает события с 14:30:00 до 14:30:10

# Использование формата nginx для --date-from
catuniq 1m \
    --date-from "20/Jan/2026:14:30:00 +0300" \
    --col-start none \
    /var/log/nginx/access.log
# Или скопировать дату прямо из лога:
catuniq 1m \
    --date-from "[20/Jan/2026:14:30:00 +0300]" \
    --col-start none \
    /var/log/nginx/access.log

# Сравнение разных временных окон
for time in "14:00:00" "15:00:00" "16:00:00"; do
    echo "=== Events at $time ==="
    catuniq 5m --date-from "2026-01-20 $time" --count /var/log/app.log | head -5
done
```

### 8. Использование переменных окружения

```bash
# Установить глобальные настройки
export CATUNIQ_DATE_TIME="1h"
export CATUNIQ_DATE_FMT='[0-9]{4\}-[0-9]{2\}-[0-9]{2\}T[0-9]{2\}:[0-9]{2\}:[0-9]{2\}'
export CATUNIQ_COL_START="none"

# Теперь можно использовать без указания этих параметров
catuniq /var/log/app.log
```

### 8. Отладка с --dry-run и --verbose

```bash
# Посмотреть какие настройки будут использованы
catuniq 10m --date-fmt='custom_pattern' --dry-run /var/log/app.log

# Включить подробный вывод для отладки
catuniq 10m --verbose /var/log/app.log 2>&1 | less
```

### 9. Удаление и замена текста (--del и --replace)

```bash
# --del: Удаление совпадающего текста (заменяет на пустую строку)
catuniq 1h \
    --col-start none \
    --date-fmt='[0-9]{4\}-[0-9]{2\}-[0-9]{2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --del 'user=[^ ]+' \
    --del 'session=[^ ]+' \
    --count --sort freq \
    /var/log/app/error.log

# Результат: "ERROR user=alice session=abc123 db_error" → "ERROR db_error"

# --replace: Замена паттерна на текст (формат: 'паттерн===>замена')
catuniq 30d \
    --col-start none \
    --date-fmt='[0-9]{4\}-[0-9]{2\}-[0-9]{2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --replace 'ERROR===>WARN' \
    --replace 'INFO===>DEBUG' \
    --count --sort freq \
    /var/log/app/error.log

# Результат: "ERROR db_timeout" → "WARN db_timeout"

# Комбинация --del и --replace для нормализации логов
catuniq 30d \
    --col-start 4 \
    --date-fmt='[0-9]{2\}/[A-Z][a-z]{2\}/[0-9]{4\}:[0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --del '[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}' \
    --replace '200===>2XX' \
    --replace '404===>4XX' \
    --replace '500===>5XX' \
    --count --sort freq \
    /var/log/nginx/access.log

# Результат:
#   IP адреса удалены
#   Коды 200 → 2XX, 404 → 4XX, 500 → 5XX
#   Позволяет видеть паттерны запросов без конкретных IP и HTTP кодов
```

### 10. Условное удаление столбцов (--remove)

```bash
# --remove: Удаляет столбец только если он совпадает с паттерном
# Формат: --remove 'номер_колонки:паттерн'

# Пример 1: Удалить HTTP код 200 из nginx логов
catuniq 30d \
    --col-start 4 \
    --date-fmt='[0-9]{2\}/[A-Z][a-z]{2\}/[0-9]{4\}:[0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --remove '9:200' \
    --count --sort freq \
    /var/log/nginx/access.log

# Результат: строки с HTTP 200 потеряют этот код, но 401, 404, 500 останутся

# Пример 2: Удалить IP адреса и HTTP коды
catuniq 30d \
    --col-start 4 \
    --date-fmt='[0-9]{2\}/[A-Z][a-z]{2\}/[0-9]{4\}:[0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --remove '1:[0-9.]+' \
    --remove '9:[0-9]+' \
    --count --sort freq \
    /var/log/nginx/access.log

# Пример 3: Удалить уровни логирования ERROR и INFO
catuniq 1h \
    --col-start none \
    --date-fmt='[0-9]{4\}-[0-9]{2\}-[0-9]{2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --remove '1:ERROR|INFO' \
    --del 'user=[^ ]+' \
    --del 'session=[^ ]+' \
    --count --sort freq \
    /var/log/app/error.log

# Результат:
#   3   db_error: connection timeout
#   2   auth_error: invalid token
#   Уровни логов (ERROR/INFO) удалены только из строк где они есть

# Разница между --cut и --remove:
# --cut: Всегда удаляет указанные колонки
# --remove: Удаляет колонку только если она совпадает с паттерном

# Пример разницы:
# --cut '9'         → Удалит колонку 9 из всех строк
# --remove '9:200'  → Удалит колонку 9 только если там значение 200
```

### 11. Условное изменение столбцов (--change)

```bash
# --change: Изменяет значение столбца только если он совпадает с паттерном
# Формат: --change 'номер_колонки:паттерн:новое_значение'
# ВАЖНО: Экранируйте двоеточия в паттерне как \:

# Пример 1: Изменить уровень логирования ERROR на WARN
catuniq 1h \
    --col-start none \
    --date-fmt='[0-9]{4\}-[0-9]{2\}-[0-9]{2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --change '1:ERROR:WARN' \
    --count --sort freq \
    /var/log/app/error.log

# Результат: все "ERROR" в первой колонке станут "WARN"

# Пример 2: Нормализация HTTP кодов в nginx логах
catuniq 30d \
    --col-start 4 \
    --date-fmt='[0-9]{2\}/[A-Z][a-z]{2\}/[0-9]{4\}:[0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --change '9:200:2XX' \
    --change '9:401:4XX' \
    --change '9:404:4XX' \
    --change '9:500:5XX' \
    --count --sort freq \
    /var/log/nginx/access.log

# Результат: HTTP коды заменены на категории (2XX, 4XX, 5XX)

# Пример 3: Использование экранированных двоеточий в паттерне
catuniq 1h \
    --col-start none \
    --date-fmt='[0-9]{4\}-[0-9]{2\}-[0-9]{2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --change '2:db_error\\::DB_ERR' \
    --change '2:auth_error\\::AUTH_ERR' \
    --del 'user=[^ ]+' \
    --count --sort freq \
    /var/log/app/error.log

# Паттерн 'db_error\:' будет искать 'db_error:' в тексте
# Результат: "db_error:" → "DB_ERR", "auth_error:" → "AUTH_ERR"

# Пример 4: Комбинация --change, --remove, --del
catuniq 30d \
    --col-start 4 \
    --date-fmt='[0-9]{2\}/[A-Z][a-z]{2\}/[0-9]{4\}:[0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    --remove '1:[0-9.]+' \
    --change '9:[0-9]+:XXX' \
    --del '"[^"]*"' \
    --count --sort freq \
    /var/log/nginx/access.log

# Результат:
#   - IP адреса удалены (--remove)
#   - Все HTTP коды заменены на XXX (--change)
#   - URL-ы удалены (--del)

# Разница между --remove, --change и --replace:
# --remove '9:200'        → Удалит колонку 9 если там 200
# --change '9:200:OK'     → Заменит значение колонки 9 с 200 на OK
# --replace '200===>OK'   → Заменит все вхождения "200" на "OK" во всей строке
```

## Форматы дат

### Nginx формат (по умолчанию)
```
[20/Jan/2026:20:55:01 +0300]
```
Паттерн (встроенный): `[[][0-9]{2\}/[A-Za-z]{3\}/[0-9]{4\}:...`

### ISO 8601 формат
```
2026-01-20T20:55:01Z
```
Паттерн: `[0-9]{4\}-[0-9]{2\}-[0-9]{2\}T[0-9]{2\}:[0-9]{2\}:[0-9]{2\}`

### Syslog формат
```
Dec 25 07:44:17
```
Паттерн: `[A-Z][a-z]{2,4\} [0-9]{1,2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}`

**Особенности:**
- Дата БЕЗ года (syslog не включает год в timestamp)
- Дата разбита на 3 колонки: `Dec` (col 1), `25` (col 2), `07:44:17` (col 3)
- Используйте `--col-start 1-3` для объединения колонок
- Автоматическое добавление года:
  - Без `--year`: автоматически пробует current, current-1, current-2
  - С `--year 2025`: использует явно указанный год

### Application log формат
```
2026-01-20 20:55:01
```
Паттерн: `[0-9]{4\}-[0-9]{2\}-[0-9]{2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}`

**Важно**: В regex паттернах используйте `{2\}` вместо `{2}` чтобы избежать bash brace expansion.

## Как это работает внутри

### Архитектура и поток обработки

catuniq использует комбинацию bash для парсинга аргументов и GNU AWK для обработки логов. Вот полная схема обработки данных:


### Ключевые особенности реализации

#### 1. Порядок применения операций (ВАЖНО!)

Операции применяются в строго определенном порядке:

```
1. --cut           → Безусловное удаление колонок
2. --remove        → Условное удаление колонок (по паттерну)
3. --change        → Условная замена значений колонок
4. --del           → Удаление текстовых паттернов
5. --replace       → Замена текстовых паттернов
```

**Почему это важно:**
- После `--remove` номера колонок сдвигаются
- `--change` работает с уже модифицированными колонками
- `--del` и `--replace` видят результат всех колоночных операций

**Пример:**
```bash
# Исходная строка: "192.168.1.1 - - [date] GET 200 1234"
# Колонки:         1            2 3   4     5   6   7

catuniq --remove '1:[0-9.]+' --change '6:200:2XX'

# После --remove '1:[0-9.]+':
#   "- - [date] GET 200 1234"
#	1  2   3     4   5   6    (сдвинулись!)
#   Колонки теперь  

# После --change '6:200:2XX':
#   "- - [date] GET 2XX 1234"   (колонка 6 теперь это "1234", не "200"!)
#   НЕПРАВИЛЬНО! Нужно было --change '5:200:2XX'
```

#### 2. Экранирование специальных символов

**Для параметра `--change` с двоеточиями:**
```bash
# Паттерн содержит двоеточие
--change '3:db_error\::DB_ERR'
#            символ ↑ экранирован как \:

# Обработка в bash:
# 1. Заменить \: на __ESCAPED_COLON__
# 2. Разделить по незаэкранированным :
# 3. Восстановить : в паттерне
```

**Для regex паттернов с backslash:**
```bash
# Паттерн: \d+ (цифры)
--del '\\d+'           # В bash нужно двойное экранирование

# Передача в AWK:
# Bash: '\\d+' → AWK получает: '\d+'
```

#### 3. Автоопределение года для syslog

Syslog даты не содержат год: `Dec 25 07:44:17`

**Алгоритм:**
1. Если `--year 2025` указан → использовать его
2. Иначе пробовать годы: current, current-1, current-2
3. Для каждого года:
   - Попытаться распарсить дату
   - Вычислить разницу с текущим временем
4. Выбрать год с минимальной разницей (ближайший к now)

**Зачем это нужно:**
- Логи от декабря 2025 читаются в январе 2026
- "Dec 25" должен стать "Dec 25 2025", не "Dec 25 2026"

#### 4. Работа с множественными колонками для даты

Syslog разбивает дату на 3 поля: `Dec` `25` `07:44:17`

**Решение:**
```bash
--col-start 1-3   # Или --col-start 1,2,3

# AWK собирает колонки:
# col_indices = [1, 2, 3]
# date_str = $1 " " $2 " " $3 = "Dec 25 07:44:17"
```

### Оптимизации производительности

1. **Единый проход по файлу**: AWK обрабатывает каждую строку один раз
2. **Ранний выход**: Строки вне временного окна пропускаются сразу (`next`)
3. **Хеш-таблицы**: `seen[]` для O(1) проверки уникальности
4. **Ленивая сортировка**: Сортировка только в конце, не на каждой строке

### Ограничения и граничные случаи

1. **Regex в AWK**: Используется ERE (Extended Regular Expressions), не PCRE
2. **Временные зоны**: Даты парсятся в локальной временной зоне системы
3. **Максимальная длина строки**: Ограничена AWK (~2048 символов по умолчанию)
4. **Память**: Все уникальные строки хранятся в памяти при `--count`

## Советы и хитрости

### 1. Удаление переменных частей

Для нахождения истинных уникальных паттернов ошибок, удаляйте:
- IP адреса: `[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}`
- User IDs: `user=[^ ]+` или `user_id=[0-9]+`
- Session IDs: `session=[a-f0-9]+`
- Request IDs: `request_id=[a-zA-Z0-9-]+`
- Timestamps в самом сообщении: `[0-9]{4\}-[0-9]{2\}-[0-9]{2\}`
- PIDs: `pid=[0-9]+`
- Порты: `:[0-9]{2,5\}`

### 2. Производительность

Для больших логов:
```bash
# Используйте zcat для сжатых логов
zcat /var/log/nginx/access.log.*.gz | catuniq 24h

# Фильтруйте перед catuniq
grep ERROR /var/log/huge.log | catuniq 1h --count
```

### 3. Колонки (--col-start)

```bash
# Дата в одной колонке (nginx access.log)
catuniq 1h --col-start 4 /var/log/nginx/access.log

# Дата разбита на несколько колонок (syslog: Dec 25 07:44:17)
catuniq 1h --col-start 1-3 /var/log/syslog

# Список конкретных колонок
catuniq 1h --col-start 1,2,3 /var/log/messages

# Смешанный формат (колонки 1, 3, 4, 5)
catuniq 1h --col-start 1,3-5 /var/log/custom.log

# Искать дату в любом месте строки (медленнее, но универсально)
catuniq 1h --col-start none /var/log/app.log
```

**Производительность col-start:**
- `--col-start N` - самый быстрый (ищет только в одной колонке)
- `--col-start 1-3` - быстро (ищет в 3 колонках)
- `--col-start none` - медленнее (ищет по всей строке)

**Когда использовать что:**
- Для syslog (`Dec 25 07:44:17`) → `--col-start 1-3`
- Для nginx access (`[20/Jan/2026:...]` в колонке 4) → `--col-start 4`
- Для JSON или произвольного формата → `--col-start none`

### 4. Комбинирование --replace

```bash
# Множественные замены применяются последовательно
catuniq 1h \
    --replace 'user=[^ ]+' \
    --replace 'ip=[0-9.]+' \
    --replace 'session=[a-f0-9]+' \
    --replace 'request_id=[^ ]+' \
    /var/log/app.log
```

### 5. Работа с --year для syslog

```bash
# Автоопределение года (пробует current, current-1, current-2)
catuniq --date-from "Dec 25 07:44:17" 1h \
    --col-start 1-3 \
    --date-fmt='[A-Z][a-z]{2,4\} [0-9]{1,2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    /var/log/syslog

# Явное указание года (для старых архивных логов)
catuniq --date-from "Dec 25 07:44:17" --year 2024 1h \
    --col-start 1-3 \
    --date-fmt='[A-Z][a-z]{2,4\} [0-9]{1,2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    /var/log/syslog.2024.gz

# Без --date-from: --year применяется к датам в логах
catuniq 1h --year 2024 \
    --col-start 1-3 \
    --date-fmt='[A-Z][a-z]{2,4\} [0-9]{1,2\} [0-9]{2\}:[0-9]{2\}:[0-9]{2\}' \
    /var/log/syslog.old
```

**Когда использовать --year:**
- ✅ Для архивных syslog логов (например, syslog.2024)
- ✅ Когда автоопределение выбирает неправильный год
- ✅ Для логов с января/февраля прошлого года
- ❌ Не нужен для логов с полной датой (ISO 8601, nginx format)
- ❌ Не нужен для свежих логов (автоопределение работает корректно)

## Отладка AWK: Уроки из разработки

Во время разработки `catuniq` мы столкнулись с несколькими интересными проблемами при работе с AWK. Вот что мы узнали:

### 1. Bash Brace Expansion vs Regex

**Проблема**: При использовании `{N}` в regex внутри bash переменных, bash может интерпретировать это как brace expansion.

```bash
# ❌ НЕ РАБОТАЕТ
pattern="\[[0-9]{2}/[A-Za-z]{3}/[0-9]{4}:[0-9]{2}:[0-9]{2}:[0-9]{2}\]"
# bash видит {2}/ и пытается раскрыть как brace expansion
# Результат: {2 исчезает, получается [0-9]/[A-Za-z]...
```

**Решение**: Экранировать слеш после закрывающей фигурной скобки:

```bash
# ✅ РАБОТАЕТ
pattern="\[[0-9]{2\}/[A-Za-z]{3\}/[0-9]{4\}:[0-9]{2\}:[0-9]{2\}:[0-9]{2\}\]"
# {2\}/ не распознается как brace expansion
```

**Как проверить**:
```bash
# Посмотреть что bash видит
echo "${VAR:-[0-9]{2}/test}"     # Вывод: [0-9]{2/test}  - слеш исчез!
echo "${VAR:-[0-9]{2\}/test}"    # Вывод: [0-9]{2}/test} - правильно!
```

### 2. Экранирование квадратных скобок в AWK regex

**Проблема**: `\[` в AWK вызывает warning и работает непредсказуемо.

```bash
# ❌ ПРЕДУПРЕЖДЕНИЕ
gawk -v pattern='\[' '{ if (match($0, pattern)) print }' file.log
# gawk: warning: escape sequence `\[' treated as plain `['
```

**Решение**: Использовать bracket expressions:

```bash
# ✅ БЕЗ ПРЕДУПРЕЖДЕНИЙ
gawk -v pattern='[[]' '{ if (match($0, pattern)) print }' file.log
# [[] - означает "символ из набора содержащего ["
# []] - означает "символ из набора содержащего ]"
```

**Полный пример**:
```bash
# Было (с warnings):
pattern='\[[0-9]{2}/[A-Za-z]{3}/[0-9]{4}:[0-9]{2}:[0-9]{2}:[0-9]{2} [+-][0-9]{4}\]'

# Стало (без warnings):
pattern='[[][0-9]{2\}/[A-Za-z]{3\}/[0-9]{4\}:[0-9]{2\}:[0-9]{2\}:[0-9]{2\} [+-][0-9]{4\}[]]'
```

### 3. Передача переменных в AWK через `-v`

**Проблема**: При передаче regex через `-v`, некоторые символы могут теряться или изменяться.

```bash
# Что мы передаем в bash
date_fmt='\[[0-9]{2}/...'

# Что получает AWK после -v "date_fmt=$date_fmt"
# Результат: [[0-9]{2/...  - backslash исчез!
```

**НЕ НУЖНО**: Удваивать backslashes если используете bracket expressions:

```bash
# ❌ ПЛОХО - двойное экранирование
date_fmt_escaped="${date_fmt//\\/\\\\}"
awk_args+=(-v "date_fmt=$date_fmt_escaped")

# ✅ ХОРОШО - используйте bracket expressions
date_fmt='[[][0-9]{2\}/...[]]'  # Без backslashes перед скобками!
awk_args+=(-v "date_fmt=$date_fmt")
```

### 4. Indirect variable reference в AWK

**Проблема**: Syntax `@variable` для indirect variable reference работает не во всех версиях AWK.

```bash
# ❌ НЕ РАБОТАЕТ в gawk 4.x
for (i = 1; i <= count; i++) {
    pattern_var = "replace_pattern_" i
    gsub(@pattern_var, "", line)  # Syntax error!
}
```

**Решение**: Использовать `SYMTAB[]` или предварительно загружать в массив:

```bash
# ✅ ВАРИАНТ 1: SYMTAB (gawk специфично)
for (i = 1; i <= count; i++) {
    gsub(SYMTAB["replace_pattern_" i], "", line)
}

# ✅ ВАРИАНТ 2: Предзагрузка в массив (более универсально)
BEGIN {
    for (i = 1; i <= replace_count; i++) {
        replace_patterns[i] = SYMTAB["replace_pattern_" i]
    }
}
# Потом в основном блоке:
for (i = 1; i <= replace_count; i++) {
    gsub(replace_patterns[i], "", line)
}
```

### 5. Отладка AWK скриптов

**Техники отладки**:

```bash
# 1. Проверить что переменная действительно передалась
gawk -v test_var="value" 'BEGIN { print "Received:", test_var }'

# 2. Сохранить переменную в файл для инспекции
BEGIN {
    print date_fmt > "/tmp/awk_debug.txt"
}

# 3. Вывести в stderr (не мешает основному выводу)
BEGIN {
    print "Debug info" > "/dev/stderr"
}

# 4. Условная отладка только для первых строк
{
    if (NR <= 5) {
        print "Line", NR, ":", $0 > "/dev/stderr"
    }
}

# 5. Отладка regex match
{
    if (match($0, pattern)) {
        print "MATCH at position", RSTART, "length", RLENGTH > "/dev/stderr"
    } else {
        print "NO MATCH" > "/dev/stderr"
    }
}
```

**Структура отладочного вывода в catuniq**:
```bash
catuniq --verbose /var/log/app.log 2>&1

# Вывод покажет:
# [DEBUG] now_epoch=..., threshold_epoch=...
# [AWK DEBUG] Starting AWK script
# [AWK DEBUG] date_fmt=...
# [AWK DEBUG] Date matched at position X length Y
# [AWK DEBUG] Line N passed time filter, timestamp: ...
```

### 6. Проверка regex паттернов

**Тестирование паттернов вне AWK**:

```bash
# Проверить через grep
echo "[20/Jan/2026:20:55:01 +0300]" | grep -P '[[][0-9]{2}/[A-Za-z]{3}/[0-9]{4}'

# Проверить в AWK отдельно
echo "[20/Jan/2026:20:55:01 +0300]" | gawk -v p='[[][0-9]{2}/[A-Za-z]{3}/[0-9]{4}' \
    '{ if (match($0, p)) print "MATCH"; else print "NO MATCH" }'

# Проверить что bash правильно раскрывает переменную
pattern='[[][0-9]{2\}/test'
echo "$pattern"  # Должно быть: [[][0-9]{2}/test

# Создать минимальный тест-кейс
cat > /tmp/test.awk <<'EOF'
BEGIN { pattern = "[[][0-9]{2}" }
{ if (match($0, pattern)) print "MATCH:", $0 }
EOF
echo "[20/Jan/2026]" | gawk -f /tmp/test.awk
```

### 7. Типичные ошибки и их решения

| Ошибка | Причина | Решение |
|--------|---------|---------|
| `{2/` вместо `{2}/` | Bash brace expansion | Использовать `{2\}/` |
| `warning: escape sequence \[` | Неправильное экранирование | Использовать `[[]` вместо `\[` |
| `syntax error: @variable` | Indirect reference | Использовать `SYMTAB[]` или массив |
| Паттерн не матчится | Backslash потерялся при передаче | Проверить через `echo "$pattern"` |
| Пустой вывод AWK | stdout может быть заблокирован | Перенаправить в файл или через `>/dev/null` проверить stderr |

### 8. Best Practices

1. **Всегда используйте одинарные кавычки для regex паттернов**:
   ```bash
   --replace 'user=[^ ]+'  # ✅ Хорошо
   --replace "user=[^ ]+"  # ❌ Может не работать
   ```

2. **Экранируйте слеши после `{N}` в bash переменных**:
   ```bash
   pattern='[0-9]{2\}/[0-9]{2\}'  # ✅ Правильно
   ```

3. **Используйте bracket expressions для `[` и `]`**:
   ```bash
   pattern='[[]date[]]'  # ✅ Вместо '\[date\]'
   ```

4. **Отлаживайте поэтапно**:
   - Сначала проверьте что bash правильно хранит переменную: `echo "$var"`
   - Потом проверьте что AWK получает правильное значение: `BEGIN {print var}`
   - Только потом тестируйте match

5. **Используйте verbose режим для production**:
   ```bash
   # Сохранить первые 50 строк лога с отладкой
   catuniq --verbose /var/log/app.log 2>&1 | head -50 > /tmp/debug.txt
   ```
